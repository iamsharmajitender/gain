## Design for Governed MCP

<div className="gain-section-lead">
  MCP standardizes how models discover tools, read resources, and invoke capabilities: but in
  enterprise it is a <strong>protocol inside architecture</strong>, not a shortcut around governance.
  G.A.I.N MCP treats every MCP server as a bounded capability surface: registered, authenticated,
  policy-gated, and fully auditable.
</div>

## MCP invocation flow

<div className="gain-section-lead">
  MCP is a <strong>transport and contract layer</strong> between intelligent clients and enterprise
  systems. The client plans; the gateway permits; the server executes: validation happens before
  results re-enter the workflow.
</div>

## G.A.I.N applied to MCP

<Details className="gain-accordion" summary="G · Grounded: What controls MCP access?">
  Grounded MCP integrations enforce authority before any tool list, resource read, or invocation
  crosses the enterprise boundary. The protocol connects systems; policy decides what is allowed.

  **Components**

  - Policy engine: allow/deny rules on servers, tools, and resources per client and tenant
  - Tool permissions: scoped manifests; sensitive tools hidden or require elevated scope
  - Human approvals: escalation for irreversible or high-risk MCP invocations
  - Guardrails: input/output filtering on MCP messages and tool arguments
  - Audit logs: record every `list_tools`, `call_tool`, and `read_resource` with actor and trace ID

  **Example:** MCP server exposes account lookup but cannot authorize transfers without approval.

  **Design questions**

  - Can this client call this tool?
  - Can it approve this action?
  - Does it need escalation?

  **Principle:** MCP exposes capability, not authority.
</Details>

<Details className="gain-accordion" summary="A · Adaptive: How MCP fits the workflow">
  Adaptive MCP integrations survive retries, session drops, and evolving server manifests.
  Orchestration owns the workflow; MCP is how capability is reached: not the workflow itself.

  **Components**

  - Orchestrator: MCP client lifecycle inside agent or control-plane runtime
  - Tool chain: ordered invocations across one or more MCP servers
  - Event bus: async tool results and notifications between services
  - State store: session context, tool outputs, and conversation state across steps
  - Retry logic: idempotent replays, timeouts, and compensating actions on failed calls

  **Example: booking flow via MCP:** Search → Validate → Reserve → Notify: each step an MCP tool
  behind policy, state carried between invocations.

  **Design questions**

  - How do clients recover from a failed MCP call?
  - How do sessions persist across reconnects?
  - How do tools chain across multiple servers?

  **Principle:** MCP is a protocol inside governed workflows.
</Details>

<Details className="gain-accordion" summary="I · Intelligent: What reasoning uses MCP?">
  Intelligent MCP clients use the model to discover and plan invocations: deterministic systems
  validate arguments, enforce policy, and execute. The model interprets tool output; it does not
  bypass entitlements.

  **Components**

  - Planning: decompose goals into MCP tool and resource steps
  - Tool selection: choose from server manifest based on task and context
  - Multi-step reasoning: iterate discover → invoke → observe until complete or blocked
  - Summarization: condense resource contents and tool results for the next planning step

  **Example:** LLM plans the next MCP tool call; the gateway executes only after policy clears.

  **Design questions**

  - What should the model decide?
  - What should be deterministic (routing, auth, validation)?

  **Principle:** Intelligence plans, systems execute.
</Details>

<Details className="gain-accordion" summary="N · Native: Where MCP runs on the platform">
  Native MCP deployments are platform-managed: servers are hosted, scaled, and observed like any
  other enterprise integration. Transport choice and gateway placement are infrastructure decisions.

  **Components**

  - Kubernetes: host MCP servers as services or sidecars with health checks and autoscaling
  - MCP gateway: central proxy for auth, routing, rate limits, and protocol normalization
  - APIs and queues: async tools, long-running jobs, and backpressure across MCP boundaries
  - Observability stack: distributed traces per MCP session, tool span, and latency SLOs

  **Design questions**

  - How does MCP traffic scale under concurrent agent sessions?
  - How is latency managed across multi-server tool chains?

  **Principle:** MCP needs platform resilience.
</Details>

## Key patterns

<Details className="gain-accordion" summary="MCP gateway">
  Front MCP traffic through a platform gateway: authenticate clients, enforce policy, normalize
  transports (stdio, SSE, streamable HTTP), and emit audit events. Product teams never connect
  directly to arbitrary MCP servers.
</Details>

<Details className="gain-accordion" summary="Server registry">
  Catalog approved MCP servers with owner, scope, environment, and manifest version. Registration is
  the control point for what enters the enterprise tool surface: same discipline as a tool registry.
</Details>

<Details className="gain-accordion" summary="Scoped manifests">
  Publish different tool and resource lists per client profile, tenant, or use case. Not every agent
  sees every capability: manifests are entitlement-aware, not all-or-nothing.
</Details>

<Details className="gain-accordion" summary="Resources vs tools">
  Use **resources** for read-only context (documents, configs, account snapshots) and **tools** for
  actions with side effects. Mixing the two blurs policy boundaries and complicates audit.
</Details>

<Details className="gain-accordion" summary="Federated MCP">
  Compose multiple domain-owned MCP servers behind one gateway. Each server owns a bounded context;
  the orchestrator chains calls: no monolithic god-server.
</Details>
